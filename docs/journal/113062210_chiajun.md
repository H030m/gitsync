**為何這樣做**

- 五人協作的硬需求是「接口先固定」。我把 Model 的 `fromMap`/`toMap`、Repository 的 stream/CRUD method 簽章、ViewModel 的 ChangeNotifier 結構、Functions 的 callable input/output、Firestore 路徑常數通通寫死成「合約」。隊員只要往對應檔案內補實作（多半是 `throw new Error('not implemented yet')` 換成真內容），不會碰到設計爭議。
- 課程 lint `prefer_initializing_formals` 跟 [課程 model pattern](../COURSE_METHODS.md#41-model--null-safe--firestore-timestamp)（`_createdAt` 私有欄位 + getter fallback）衝突，所以在 `analysis_options.yaml` 統一關掉這個 rule（決策也寫進 MEMORY）。
- `breakdownTask` handler 的**分散式鎖直接寫好**而非留 stub，因為 [`ARCHITECTURE.md §5.1`](../ARCHITECTURE.md#51-flow-1--breakdowntaskflow任務拆解) 強調這是「兩個成員同時點會直接爛掉」的關鍵防護，不能留給隊員「之後再說」。`scheduledUnstickBreakdown` 兜底排程也完整寫好。
- secrets 採「root `secrets/` + `functions/.secret.local` 兩層」是因為：root 那層人類 ↔ AI 看得到，是「中央事實來源」；`functions/.secret.local` 是 Firebase emulator 慣例（部署不用，emulator 才用），兩邊都列在 `.gitignore`。
- `firebase_options.dart` 故意留成 throws UnsupportedError 的 placeholder——讓 `flutter analyze` 過但 runtime 會在啟動時提示「請跑 `flutterfire configure`」。實際接 Firebase 專案後 flutterfire CLI 會自動覆寫。
- TODO 在 view 跟 flow 大量留下，但**每個 TODO 都點名對應的 Sprint 與 ARCHITECTURE 章節**，例如 `// TODO Sprint 4: parse #N from commit message, compute embedding...`，讓接手的人不用回頭翻文件。
